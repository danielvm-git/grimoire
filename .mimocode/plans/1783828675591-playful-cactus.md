# Refactor Plan: Consolidate GitHub Actions Workflows

## Problem Statement

Grimoire has 4 separate GitHub Actions workflows (`ci.yml`, `deploy.yml`, `docs.yaml`, `codeql.yml`) that were hand-authored and diverge from the fleet-wide templates in `danielvm-git/.github`. The template repo has consolidated 16 templates into 9 unified pipelines. Grimoire needs migration to stay aligned with the fleet.

The current CI also has gaps: `ruff format --check` is not enforced in CI (only locally via `just check`), and there is no automated release/versioning.

## Solution

Replace the 4 old workflows with 2 template-based workflows:

1. **`ci-cd-python.yml`** — 4-stage pipeline: `ci → verify → semantic-release → deploy` + CodeQL
2. **`ci-cd-pages-mkdocs.yml`** — MkDocs build + GitHub Pages deploy

Key adaptations from the standard template:
- Use **pyright** instead of mypy (matches project convention)
- Include **codespell** and **actionlint** in CI (from current `just lint`)
- Add **`ruff format --check`** to CI (currently missing from CI)
- Enable **semantic-release** for automated versioning
- Call tools **directly** (not via `just`)

### 5. `feat(ci): add ci-cd-python.yml — CI and CodeQL stages`

**Changes:** Create `.github/workflows/ci-cd-python.yml` with `ci` and `codeql` jobs only.

**CI job** (single job, runs tools directly):
```
checkout → setup-python 3.13 → pip install uv → uv sync --group dev
→ uv run ruff check .
→ uv run ruff format --check
→ uv run pyright src
→ uv run codespell src
→ uv run pytest -q
→ uv run python -m build

```

**CodeQL job** (parallel with ci):
```
checkout → codeql-action/init (python) → codeql-action/analyze
```

**Triggers:** push/PR to main (paths-ignore docs, *.md), workflow_dispatch

**Concurrency:** `ci-python-${{ github.ref }}`, cancel-in-progress: true

**Verify:** Push branch, open PR — both `ci` and `codeql` jobs should appear and run

### 6. `feat(ci): add verify and semantic-release stages`

**Changes:** Add `verify` and `semantic-release` jobs to `ci-cd-python.yml`. Create `.releaserc.json` and minimal `pyproject.toml`.

**verify job** (gate):
- `needs: [ci, codeql]` — blocks release unless both passed
- Single step: confirms all quality gates passed

**semantic-release job** (main only):
- `needs: [verify]`
- Condition: `github.event_name == 'push' && github.ref == 'refs/heads/main'`
- Node.js 22 + npm semantic-release with `@semantic-release/changelog` + `@semantic-release/git`
- Permissions: `contents: write`, `issues: write`, `pull-requests: write`

**`.releaserc.json`:**
```json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    "@semantic-release/changelog",
    ["@semantic-release/git", {
      "assets": ["CHANGELOG.md"],
      "message": "chore(release): ${nextRelease.version}\n\n${nextRelease.notes}"
    }],
    "@semantic-release/github"
  ]
}
```

**`pyproject.toml`** (minimal, for semantic-release + uv):
```toml
[project]
name = "grimoire-dashboard"
version = "0.1.8"
requires-python = ">=3.13"
```

**Verify:** Push branch — `verify` job should appear in PR checks, `semantic-release` should be skipped on PR (correct — only runs on main push)

## Decision Document

### Modules modified
- `.github/workflows/` — all 4 old files deleted, 2 new files created

### New files
- `.releaserc.json` — semantic-release configuration
- `pyproject.toml` — minimal project metadata for semantic-release version management

### Unchanged files
- `setup.py` — stays as-is (BigBase uses it for `pip install`)
- `Justfile` — stays as-is (local dev still uses `just` commands)
- Source code, tests, configs — no changes

### Technical clarifications
- pyright replaces mypy from the template (user decision)
- codespell + actionlint added to CI (from current `just lint`, not in template)
- `ruff format --check` added to CI (gap in current CI — was only enforced locally)
- `uv run python -m build` added to CI (artifact upload for potential future use)
- semantic-release manages version in `pyproject.toml`, not `setup.py` (setup.py version is for BigBase only)
- Weekly CodeQL schedule dropped — can be added as a separate minimal workflow later if needed
- `vulture` (dead code detection) dropped from CI — it's slow and produces noisy output; keep it in `just lint` for local use

### Architectural decisions
- Single CI job (not separate lint/test) — faster, simpler, one Python setup cycle
- Tools called directly in CI — no `just` dependency, explicit steps
- Conventional Commits enforced by verify job — all commits to main must follow the format
- Co-authored-by AI attribution blocked by verify job

## Testing Decisions

- The workflow YAML files themselves are the deliverable — validate with `yamllint` or `actionlint`
- After pushing the branch, the PR checks will prove the CI job works (ruff, pyright, codespell, pytest all pass)
- No new unit tests needed — this is infrastructure, not application code
- Prior art: the existing `just check` command validates the same tool chain locally

## Out of Scope

- **Weekly CodeQL schedule** — requires a `schedule` trigger which doesn't compose cleanly with `paths-ignore`. Can add a separate `codeql-schedule.yml` later.
- **Migrating from `setup.py` to full `pyproject.toml`** — separate refactor. The minimal `pyproject.toml` here is only for semantic-release.
- **Docker/PyPI publish in CI** — manual-only via `just publish-docker` / `just publish`. Not part of consolidation.
- **Caching uv/pip dependencies** — performance improvement, not consolidation. Can add with `actions/cache` later.
- **Matrix testing (multiple Python versions)** — project pins 3.13. Separate concern.
- **Dependabot/Renovate** — separate concern.

## Secrets Required

## Bug Fix: pytest-asyncio Configuration

### Problem

CI run `29204857070` fails with 183 errors and 72 failures. Root cause: `pytest-asyncio` is not configured with `asyncio_mode = "auto"`, so async test methods and async fixtures (like `async_client` in `tests/conftest.py`) are not handled by pytest.

Error message from CI:
```
'test_list_actions' requested an async fixture 'action_client', with no plugin or hook that handled it.
```

### Fix

Add `asyncio_mode = "auto"` to `[tool.pytest.ini_options]` in `pyproject.toml` (line 49).

**File:** `pyproject.toml`

```toml
[tool.pytest.ini_options]
pythonpath = ["src"]
asyncio_mode = "auto"
```
